# 깃헙 공동협업 가이드

이 문서는 팀장과 팀원이 같은 GitHub 저장소를 함께 사용할 때 필요한 기본 협업 흐름을 정리한 문서입니다.

원문 흐름은 유지하되, 실제로 바로 사용할 수 있게 아래처럼 정리했습니다.

- 기본 브랜치 이름은 `main` 기준으로 설명합니다.
- 새 브랜치를 만들 때는 `git branch -b`가 아니라 `git switch -c` 또는 `git checkout -b`를 사용합니다.
- 작업 전에 항상 `pull`로 최신 변경 사항을 먼저 받아오는 것을 기준으로 설명합니다.

## 1. 협업 시작 전 준비

팀장은 GitHub 저장소에 공동 작업자를 초대하고, 팀원들은 초대를 수락합니다.

그다음 팀장과 팀원 모두 각자 컴퓨터에 작업용 저장소를 준비해야 합니다.

## 2. 각 작업자 PC에서 지역 저장소 만들기

새 프로젝트를 직접 시작하는 사람은 지역 저장소를 만들고, Git 사용자 정보를 설정합니다.

```bash
git init projecttest
cd projecttest
git config user.name "사용자 이름"
git config user.email "메일 주소"
```

설명:

- `git init projecttest`: `projecttest` 폴더를 Git 저장소로 만듭니다.
- `git config user.name`: 커밋 작성자 이름을 설정합니다.
- `git config user.email`: 커밋 작성자 이메일을 설정합니다.

참고:

- 이 설정은 저장소별로 따로 할 수도 있고, `--global`을 붙여 전체 PC 공통값으로 설정할 수도 있습니다.

## 3. 팀장이 원격 저장소에 첫 commit, push 하기

팀장은 먼저 지역 저장소와 GitHub 원격 저장소를 연결한 뒤, 첫 파일을 커밋하고 푸시합니다.

```bash
git remote add origin 원격저장소URL
vim viewtest.txt
git add .
git commit -m "viewtest"
git push -u origin main
```

설명:

- `git remote add origin`: 현재 로컬 저장소와 GitHub 저장소를 연결합니다.
- `git add .`: 변경 파일을 스테이징합니다.
- `git commit -m`: 변경 내용을 하나의 커밋으로 기록합니다.
- `git push -u origin main`: `main` 브랜치를 원격 저장소에 처음 업로드합니다.

첫 푸시가 끝나면 GitHub 저장소에서 파일이 올라간 것을 확인할 수 있습니다.

## 4. 공동 작업자는 원격 저장소를 clone 하기---------------팀원은 여기서부터

팀원이 기존 GitHub 저장소에서 협업을 시작할 때는 새로 `git init` 하지 않고, 원격 저장소를 그대로 복제합니다.

```bash
git clone 원격저장소URL
cd 저장소이름
```

설명:

- `git clone`: 원격 저장소의 파일, 커밋 기록, 브랜치 정보를 로컬로 복제합니다.

## 5. 첫 작업이 아니라면 먼저 pull 하기

협업에서는 누군가 먼저 수정해서 push했을 가능성이 높습니다.
그래서 작업을 시작하기 전에는 항상 최신 변경 사항을 먼저 받아와야 합니다.

```bash
git pull origin main
```

왜 필요한가:

- 내 PC에는 없는 최신 커밋이 원격 저장소에 올라와 있을 수 있습니다.
- 이 상태에서 바로 push하면 `rejected` 오류가 발생할 수 있습니다.

예시 흐름:

1. 팀장이 `viewtest.txt`를 수정해서 push합니다.
2. 팀원 1은 예전 상태를 기준으로 다른 작업을 합니다.
3. 팀원 1이 바로 push하면 원격 저장소의 최신 커밋이 없어 충돌 또는 거절이 날 수 있습니다.
4. 이때 먼저 `git pull origin main`으로 최신 내용을 받아와야 합니다.
5. pull 후 다시 commit 또는 push를 진행합니다.

작업 예시:

```bash
vim viewtest02.txt
git add viewtest02.txt
git commit -m "viewtest02"
git pull origin main
git push -u origin main
```

참고:

- `git pull` 과정에서 자동 병합이 일어나면 병합 커밋 메시지 입력 창이 열릴 수 있습니다.
- 충돌이 나면 파일을 직접 수정한 뒤 다시 `add`, `commit` 해야 합니다.

## 6. 기능별로 새 브랜치를 만들어 작업하기

협업에서는 보통 각 작업자 또는 기능별로 브랜치를 따로 만들어 작업합니다.
이렇게 해야 `main` 브랜치에 바로 섞이지 않고 안전하게 변경 사항을 관리할 수 있습니다.

### 6-1. 먼저 최신 내용 받기

```bash
git pull origin main
```

### 6-2. 새 브랜치 만들고 이동하기

```bash
git switch -c team1
```

또는 구버전 Git에서는:

```bash
git checkout -b team1
```

설명:

- `team1`이라는 새 브랜치를 만들고 그 브랜치로 바로 이동합니다.

### 6-3. 작업 후 commit 하기

```bash
vim viewtest03.txt
git add viewtest03.txt
git commit -m "viewtest03"
```

### 6-4. 새 브랜치를 원격 저장소에 push 하기

```bash
git push -u origin team1
```

설명:

- `origin team1`은 원격 저장소에 `team1` 브랜치를 새로 만들면서 업로드한다는 뜻입니다.

푸시가 끝나면 GitHub 저장소의 브랜치 목록에서 새 브랜치를 확인할 수 있습니다.

## 7. Pull Request로 브랜치 병합하기

브랜치에 작업한 내용은 바로 `main`에 반영되지 않습니다.
GitHub에서 Pull Request를 만들어 검토 후 병합해야 `main`에 들어갑니다.

기본 흐름:

1. GitHub 저장소에서 방금 푸시한 브랜치를 확인합니다.
2. `New pull request`를 클릭합니다.
3. 변경 내용 설명을 작성하고 `Create pull request`를 누릅니다.
4. 공동 작업자가 내용을 검토합니다.
5. 문제가 없으면 `Merge pull request`를 눌러 병합합니다.
6. `Confirm merge`를 누르면 병합이 완료됩니다.
7. 병합 후 `main` 브랜치에서 파일이 반영된 것을 확인합니다.

참고:

- PR에서는 변경 내용 설명, 질문, 리뷰 요청 등을 남길 수 있습니다.
- 누가 병합했는지도 GitHub에서 확인할 수 있습니다.

## 8. 협업할 때 꼭 기억할 규칙

- 작업 시작 전에 항상 `git pull origin main`을 먼저 합니다.
- 기능 개발은 가능하면 `main`이 아니라 새 브랜치에서 진행합니다.
- 작업이 끝나면 `git push -u origin 브랜치이름`으로 올립니다.
- 최종 반영은 Pull Request로 진행합니다.
- 충돌이 나면 당황하지 말고 최신 내용을 받은 뒤 수정하고 다시 커밋합니다.

## 9. 자주 쓰는 명령어 모음

### 새 저장소 시작

```bash
git init projecttest
cd projecttest
git config user.name "사용자 이름"
git config user.email "메일 주소"
```

### 원격 저장소 연결 후 첫 push

```bash
git remote add origin 원격저장소URL
git add .
git commit -m "첫 커밋"
git push -u origin main
```

### 원격 저장소 복제

```bash
git clone 원격저장소URL
cd 저장소이름
```

### 작업 전 최신 내용 받기

```bash
git pull origin main
```

### 새 브랜치에서 작업하기

```bash
git switch -c team1
git add .
git commit -m "작업 내용"
git push -u origin team1
```

## 10. 한 줄 요약

GitHub 협업의 기본은 아래 순서입니다.

`clone 또는 init -> pull -> 작업 -> commit -> push -> pull request -> merge`
